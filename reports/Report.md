# Отчет о проведенном тестировании

## 1. Краткое описание

Было проведено автоматизированное тестирование веб-сервиса «Путешествие дня», предназначенного для покупки тура с
оплатой банковской картой и оформлением кредита.

Сервис взаимодействует с банковским API и СУБД (MySQL / PostgreSQL), в которой фиксируются результаты операций.

В рамках тестирования были реализованы и выполнены:

- UI автотесты (Selenide, Page Object Model)
- API тесты (REST Assured)
- проверки состояния базы данных (SQL-валидация)

---

## 2. Тестовые данные

В тестировании использовались:

- валидные и невалидные номера банковских карт (предоставлены в рамках задания)
- граничные и некорректные значения (генерировались с использованием Java DataGenerator)
- остальные поля (месяц, год, владелец, CVC) формировались автоматически в рамках тестов

## 6. Анализ дефектов

### API тестирование

Из 11 API тестов 2 завершились успешно, 9 — с ошибками.

Выявленные проблемы:

- некорректная обработка невалидных входных данных на стороне backend
- отсутствие единообразного формата ошибок в ответах API
- расхождения между фактическими ответами и ожидаемым контрактом
- в отдельных случаях используется некорректный HTTP-статус (отсутствие стандартизации 4xx ответов)

Данные проблемы указывают на нарушение контрактной модели API и отсутствие централизованной обработки ошибок.

---

### UI тестирование

Из 27 UI тестов 11 завершились успешно, 16 — с ошибками.

Выявленные проблемы:

- отсутствие строгой клиентской валидации форм
- возможность отправки формы с невалидными данными
- нестабильное отображение сообщений об ошибках
- несогласованность состояния UI и ответа backend

Данные дефекты указывают на отсутствие централизованной логики валидации и управления состоянием формы.

## 8. Рекомендации по устранению дефектов

### Backend (API слой)

- реализовать строгую серверную валидацию входных данных с возвратом стандартизированных ошибок
- привести API к единому контракту (единый формат response / error schema)
- обеспечить корректное использование HTTP статус-кодов (2xx / 4xx / 5xx)
- внедрить централизованную обработку ошибок

### Frontend (UI слой)

- внедрить клиентскую валидацию форм до отправки данных на сервер
- унифицировать отображение сообщений об ошибках
- обеспечить блокировку отправки формы при наличии невалидных данных
- добавить стабильные селекторы (data-test-id) для автоматизации тестирования

### Архитектура системы

- разделить бизнес-логику и слой представления
- обеспечить согласованность UI и API контрактов
- стандартизировать структуру данных между сервисами

### Тестируемость системы

- добавить серверное логирование запросов и ответов
- обеспечить стабильность тестового окружения для автотестов
- подготовить и поддерживать актуальную техническую документацию по API и БД

---

## 9. Общий вывод

В ходе тестирования выявлены критические и значительные дефекты, влияющие на корректность ключевых бизнес-сценариев
оформления и оплаты туров.

Основные проблемы связаны с:

- нарушением контрактного взаимодействия между frontend и backend
- отсутствием единых стандартов обработки ошибок
- недостаточной валидацией пользовательских данных
- несогласованностью UI и API уровней системы

Текущее состояние системы требует доработки backend и frontend компонентов для обеспечения стабильной и предсказуемой
работы пользовательских сценариев.
