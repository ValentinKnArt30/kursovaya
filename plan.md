### Позитивные сценарии

#### Сценарий 1: Оплата тура дебетовой картой (успешный платёж)

**Входные данные:**
* номер карты: `1111 2222 3333 4444` (APPROVED);
* срок действия: `12/28`;
* CVC: `123`;
* имя владельца: `IVAN IVANOV`.

**Шаги:**
1. Открыть страницу оплаты тура.
2. Ввести номер карты.
3. Заполнить срок действия.
4. Ввести CVC.
5. Ввести имя и фамилию владельца карты.
6. Нажать кнопку «Продолжить».
7. Дождаться завершения операции.

**Ожидаемые результаты:**

* **UI:** сообщение «Успешно, Операция одобрена банком» (зелёного цвета), перенаправление на страницу подтверждения
  заказа.
* **СУБД:**
    * в таблице `order_entity` создана запись с `payment_id`;
    * в таблице `payment_entity` создана запись:
        * `transaction_id = payment_id` (из `order_entity`);
        * `status = 'APPROVED'`;
        * `amount` соответствует стоимости тура;
        * `created_at` — актуальная дата и время;
        * поля с данными карты — `NULL`.

#### Сценарий 24: Проверка повторного использования APPROVED‑карты

**Входные данные:** APPROVED‑карта `1111 2222 3333 4444`.

    * в `payment_entity` создана запись с новым `transaction_id`, статус `APPROVED`.

#### Сценарий 25: Проверка очистки формы после успешной операции

**Шаги:**

#### Сценарий 26: Проверка обработки прерывания платежа

## 3. Перечень и описание возможных рисков при автоматизации

| Риск                                             | Описание                                                                                   | Меры по минимизации                                                                                                  |
|--------------------------------------------------|--------------------------------------------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------|
| **Недоступность симулятора банковских сервисов** | Сервис может быть временно недоступен или перегружен из‑за активности других студентов.    | Планировать тестирование в периоды низкой нагрузки. В тестах предусмотреть повторные попытки запросов (retry logic). |
| **Изменения в UI приложения**                    | Элементы интерфейса могут измениться, что приведёт к падению UI‑тестов.                    | Использовать паттерн Page Object Model для изоляции локаторов. Регулярно актуализировать локаторы.                   |
| **Проблемы с подключением к СУБД**               | Ошибки конфигурации или недоступность БД могут блокировать тесты, проверяющие записи в БД. | Автоматизировать проверку подключения к БД перед запуском тестов. Использовать Docker для развёртывания тестовых БД. |
| **Ложные срабатывания из‑за задержек**           | Задержки в ответах сервисов могут приводить к таймаутам.                                   | Увеличить таймауты в тестах. Добавить механизмы ожидания (explicit waits) для UI‑элементов в Selenide.               |
| **Ограничения по ресурсам**                      | Недостаток вычислительных ресурсов (память, CPU) при запуске контейнеров и тестов.         | Оптимизировать конфигурацию Docker. Запускать тесты последовательно, а не параллельно.                               |
| **Ошибки в тестовых данных**                     | Некорректные тестовые данные (карты, сроки) могут исказить результаты.                     | Использовать заранее проверенные тестовые данные. Валидировать входные данные перед тестом.                          |
| **Проблемы совместимости версий**                | Несовместимость версий инструментов (Selenide, JUnit, Gradle) может вызвать ошибки сборки. | Фиксировать версии зависимостей в `build.gradle`. Использовать Docker‑образы с предустановленным окружением.         |
