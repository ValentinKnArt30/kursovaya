## 1. Перечень автоматизируемых сценариев с шагами и ожидаемыми результатами


### Позитивные сценарии


#### Сценарий 1: Оплата тура дебетовой картой (успешный платёж)


**Шаги:**
1. Открыть страницу оплаты тура.
2. Ввести номер APPROVED‑карты: `1111 2222 3333 4444`.
3. Заполнить срок действия (например, 12/28).
4. Ввести CVC (например, 123).
5. Нажать кнопку «Оплатить».
6. Дождаться завершения операции.

**Ожидаемые результаты:**
* **UI:** сообщение «Оплата прошла успешно» (зелёного цвета), перенаправление на страницу подтверждения заказа с деталями тура.
* **API:** в логах симулятора — запрос к `Payment Gate` (метод `POST`), статус ответа `200 OK`, тело ответа содержит `status: "APPROVED"`.
* **СУБД:**
    * в таблице `order_entity` создана запись с `payment_id`;
    * в таблице `payment_entity` создана запись со следующими полями:
        * `transaction_id = payment_id` (из `order_entity`);
        * `status = 'APPROVED'`;
        * `amount` соответствует стоимости тура;
        * `created_at` — актуальная дата и время;
        * поля с данными карты — `NULL`.
* **Отчётность:** тест в Allure — «passed», прикреплены скриншоты каждого шага, отчёт Gradle показывает статус «успешно».

#### Сценарий 2: Оформление кредита на тур (успешное одобрение)

**Шаги:**
1. Открыть форму оформления кредита.
2. Ввести номер APPROVED‑карты: `1111 2222 3333 4444`.
3. Заполнить срок действия и CVC.
4. Нажать кнопку «Оформить кредит».
5. Дождаться ответа от кредитного сервиса.

**Ожидаемые результаты:**
* **UI:** сообщение «Кредит одобрен» (зелёного цвета), отображение деталей кредитного договора (сумма, срок, ставка).
* **API:** запрос к `Credit Gate` (метод `POST`), статус `200 OK`, ответ содержит `status: "APPROVED"` и параметры кредита (`loan_amount`, `term_months`, `interest_rate`).
* **СУБД:**
    * в таблице `order_entity` создана запись с `credit_id`;
    * в таблице `credit_request_entity` создана запись:
        * `bank_id = credit_id` (из `order_entity`);
        * `status = 'APPROVED'`;
        * `loan_terms` содержит JSON с параметрами кредита;
        * `created_at` — актуальная дата;
        * поля карты — `NULL`.
* **Отчётность:** тест в Allure — «passed», скриншот договора, отчёт Gradle подтверждает успешное выполнение.

### Негативные сценарии


#### Сценарий 3: Отказ в оплате дебетовой картой

**Шаги:**
1. Открыть страницу оплаты.
2. Ввести номер DECLINED‑карты: `5555 6666 7777 8888`.
3. Заполнить остальные поля формы.
4. Нажать «Оплатить».
5. Дождаться ответа.

**Ожидаемые результаты:**
* **UI:** сообщение «Ошибка: платёж отклонён банком» (красного цвета), форма остаётся активной, перенаправления нет.
* **API:** запрос к `Payment Gate`, ответ `402 Payment Required` или `200 OK` с `status: "DECLINED"`.
* **СУБД:** запись в `payment_entity` со статусом `DECLINED`, остальные поля (кроме `id` и `created_at`) — `NULL`, сумма тура указана.
* **Отчётность:** тест — «passed» в Allure, баг‑репорт, если сообщение неинформативно или интерфейс «зависает».

#### Сценарий 4: Отказ в выдаче кредита

**Шаги:**
1. Открыть форму кредита.
2. Ввести DECLINED‑карту (`5555 6666 7777 8888`).
3. Заполнить форму и отправить заявку.
4. Дождаться ответа сервиса.

**Ожидаемые результаты:**
* **UI:** сообщение «Банк отказал в выдаче кредита» (красного цвета), возможные причины (например, «Недостаточный доход»), форма доступна для повторной подачи.
* **API:** запрос к `Credit Gate`, ответ с `status: "DECLINED"`, `rejection_reason: "INSUFFICIENT_INCOME"` или статус `403 Forbidden`.
* **СУБД:** запись в `credit_request_entity`:
    * `status = 'DECLINED'`;
    * `rejection_reason = 'INSUFFICIENT_INCOME'`;
    * `created_at` — корректная дата;
    * поля карты — `NULL`.
* **Отчётность:** тест — «passed» в Allure, баг, если текст отказа нечитаем.

#### Сценарий 5: Некорректные данные карты

**Шаги:**
1. Открыть любую форму оплаты.
2. Ввести несуществующий номер (например, `1234 5678 9012 3456`).
3. Попытаться отправить форму.

**Ожидаемые результаты:**
* **UI:** подсказка «Неверный номер карты» (красного цвета) под полем, кнопка неактивна, фокус на поле ошибки.
* **API:** запроса нет (валидация на клиенте).
* **СУБД:** записей не создано (`SELECT COUNT(*) FROM payment_entity WHERE created_at > NOW() - INTERVAL 1 MINUTE` → `0`).
* **Отчётность:** тест — «passed» в Allure, баг, если подсказка отсутствует или запрос всё же отправлен.

* **API:** запроса нет.

* **Отчётность:** тест — «passed» в Allure и Gradle, баг, если сообщение некорректно или запрос уходит.

## 2. Перечень используемых инструментов с обоснованием выбора


| Инструмент | Назначение | Обоснование выбора |
|----------|-----------|-----------------|
| **Selenide** | UI‑тестирование веб‑интерфейса | Упрощает работу с Selenium WebDriver: встроенные ожидания, лаконичный синтаксис, автоматическая очистка ресурсов. Идеален для быстрого написания стабильных UI‑тестов. |
| **JUnit 5** | Фреймворк для написания и запуска тестов | Современная версия JUnit с улучшенной поддержкой параметризованных тестов, расширений и аннотаций. Интегрируется с Selenide и Allure. |
| **RestAssured** | Проверка API‑запросов | Удобный DSL для тестирования REST‑сервисов на Java. Позволяет легко проверять статусы, тела ответов и заголовки. |
| **Allure** | Генерация отчётов по результатам тестов | Наглядные интерактивные отчёты с детализацией по шагам, скриншотами, логами и видео. Помогает быстро анализировать результаты. |
| **Gradle** | Сборка проекта и управление зависимостями | Популярный инструмент сборки для Java‑проектов. Обеспечивает гибкость конфигурации, интеграцию с CI/CD и управление зависимостями. |
| **PostgreSQL/MySQL** | Тестовые базы данных | Соответствие требованиям задания. Возможность проверки работы приложения с разными СУБД. |
| **Docker** | Контейнеризация окружения | Изоляция тестового окружения, упрощение настройки и запуска СУБД и приложения. Позволяет воспроизвести production‑подобную среду. |
| **Git/GitHub** | Контроль версий и размещение репозитория | Стандарт индустрии. Поддержка совместной работы, CI/CD‑интеграций и удобного обзора кода. |

## 3. Перечень и описание возможных рисков при автоматизации


| Риск | Описание | Меры по минимизации |
|------|---------|-------------------|
| **Недоступность симулятора банковских сервисов** | Сервис может быть временно недоступен или перегружен из‑за активности других студентов. | Планировать тестирование в периоды низкой нагрузки. В тестах предусмотреть повторные попытки запросов (retry logic). |
| **Изменения в UI приложения** | Элементы интерфейса могут измениться, что приведёт к падению UI‑тестов. | Использовать паттерн Page Object Model для изоляции локаторов. Регулярно актуализировать локаторы. |
| **Проблемы с подключением к СУБД** | Ошибки конфигурации или недоступность БД могут блокировать тесты, проверяющие записи в БД. | Автоматизировать проверку подключения к БД перед запуском тестов. Использовать Docker для развёртывания тестовых БД. |
| **Ложные срабатывания из‑за задержек** | Задержки в ответах сервисов могут приводить к таймаутам. | Увеличить таймауты в тестах. Добавить механизмы ожидания (explicit waits) для UI‑элементов в Selenide. |
| **Ограничения по ресурсам** | Недостаток вычислительных ресурсов (память, CPU) при запуске контейнеров и тестов. | Оптимизировать конфигурацию Docker. Запускать тесты последовательно, а не параллельно. |
| **Ошибки в тестовых данных** | Некорректные тестовые данные (карты, сроки) могут исказить результаты. | Использовать заранее проверенные тестовые данные. Валидировать входные данные перед тестом. |
| **Проблемы совместимости версий** | Несовместимость версий инструментов (Selenide, JUnit, Gradle) может вызвать ошибки сборки. | Фиксировать версии зависимостей в `build.gradle`. Использовать Docker‑образы с предустановленным окружением. |
